# 一致性变更管理规范

**版本**: V1.0
**日期**: 2026-04-01
**标题**: PRD/Tech/代码一致性变更管理规范
**状态**: 已迁移

---

## 一、问题背景

PRD 061 实施时暴露的问题：
1. PM 跳过 CTO 直接让 Engineer 做变更（删除了 `issue-trigger.yml`）
2. 漏掉了 `post-comment.yml` 需要保留的需求
3. Engineer 未验证 PRD/Tech/代码一致性

根因：缺乏变更管理规范，PM 可直接指派 Engineer，Engineer 无一致性检查要求。

---

## 二、变更等级定义

| 等级 | 定义 | Tech同步 | Gate评审 | 版本号规则 |
|------|------|---------|---------|-----------|
| **S (小变更)** | 仅修改PRD文字描述、格式调整、错别字修正 | 无需 | 无需 | v1.0 → v1.1 |
| **M (中变更)** | 修改功能细节、验收标准调整、优先级调整 | 局部同步 | 局部评审 | v1.0 → v1.2 |
| **L (大变更)** | 影响系统架构、模块边界、数据模型、外部接口 | 全面同步 | 完整Gate评审 | v1.0 → v2.0 |

### 变更等级判定矩阵

```
问题: 本次变更是否影响以下维度?

架构设计 - 是 → 大变更 (L)
模块边界 - 是 → 大变更 (L)
数据模型 - 是 → 大变更 (L)
外部接口 - 是 → 大变更 (L)

功能细节 - 是 → 中变更 (M)
验收标准 - 是 → 中变更 (M)
优先级 - 是 → 中变更 (M)
非功能需求(性能要求) - 是 → 中变更 (M)

文字描述 - 是 → 小变更 (S)
格式调整 - 是 → 小变更 (S)
错别字修正 - 是 → 小变更 (S)

判定原则: 从高优先级判定，以第一个满足的条件为准
```

## 三、变更管理流程

### 3.1 变更流程总览

```
变更触发 → PM判定变更等级
              │
    ┌─────────┼─────────┐
    │         │         │
    ▼         ▼         ▼
  S (小)    M (中)    L (大)
    │         │         │
    │         ▼         ▼
    │    Tech局部更新   完整Gate评审
    │         │         │
    ▼         ▼         ▼
  版本记录  Tech同步    Tech全面更新
              │         │
              └────┬────┘
                   ▼
              Engineer实施
```

### 3.2 小变更 (S) 流程

```
适用范围: 仅修改PRD文字描述、格式调整、错别字修正，不影响功能实现

流程:
1. PM更新文档版本号 (v1.0 → v1.1)
2. 在"版本更新记录"中添加变更记录
3. git push

版本更新记录格式:
| 版本 | 日期 | 更新内容 | 更新人 |
|------|------|----------|--------|
| v1.0 | 初始版本 | 初始版本 | PM |
| v1.1 | YYYY-MM-DD | [变更描述] | PM |
```

### 3.3 中变更 (M) 流程

```
适用范围: 修改功能细节、验收标准调整，不影响架构和模块边界

流程:
1. PM更新PRD版本号 (v1.0 → v1.2)
2. PM通知CTO需要进行Tech局部更新
3. CTO评审变更范围，确认局部更新方案
4. CTO更新Tech相关章节（架构/模块/接口中受影响的部分）
5. Tech版本号同步更新
6. PM确认Tech更新内容
7. 局部评审通过记录
```

### 3.4 大变更 (L) 流程

```
适用范围: 影响架构设计、模块边界、数据模型，需要重新评估技术可行性

流程:
1. PM更新PRD版本号 (v1.0 → v2.0)
2. PM发起完整Gate评审
3. CTO进行技术可行性评审（Gate 1）
4. CTO更新Tech文档（如需要）
5. PM确认需求覆盖（Gate 2）
6. 相关评审人确认风险（如适用）
7. Engineer从TaskList领取任务

大变更禁止行为:
❌ PM不得跳过Gate评审直接指派Engineer
❌ Engineer不得在Gate评审通过前开始开发
```

## 四、Tech文档同步机制

### 4.1 PRD变更触发Tech更新的条件

| PRD变更类型 | Tech同步要求 | 时限 | 评审要求 |
|------------|-------------|------|---------|
| 小变更 (S) | 无需同步 | - | - |
| 中变更 (M) | 局部同步 | 变更确认后48h内 | CTO确认 + PM确认 |
| 大变更 (L) | 全面同步 | 变更确认后72h内 | 完整Gate评审 |

### 4.2 Tech同步检查清单

```
CTO触发Tech同步时检查:
□ 确认PRD变更内容
□ 识别受影响的Tech章节
□ 评估是否需要新增/删除/修改模块
□ 确认接口是否受影响
□ 更新Tech版本号
□ 更新Tech评审记录

Tech同步完成标准:
□ PRD每个功能变更都有对应的Tech更新
□ Tech更新内容经过评审（如需）
□ Tech版本号与PRD版本号对应
□ 相关文档已git push
```

### 4.3 重新Gate评审的触发条件

满足以下任一条件时，必须重新进行Gate评审：

| 条件 | 说明 | 评审类型 |
|------|------|---------|
| 架构变更 | 影响系统架构、模块边界 | 必须完整Gate |
| 接口变更 | 外部API接口变化 | 必须完整Gate |
| 数据模型变更 | 数据库Schema变化 | 必须完整Gate |
| 安全需求变更 | 新增安全要求 | 必须完整Gate |
| 性能要求重大调整 | 性能指标大幅提高 | 必须完整Gate |

## 五、Engineer一致性检查

### 5.1 Engineer实施前必须验证

```
Engineer实施前一致性检查清单 V1.0

□ PRD版本与Tech版本一致
   └─ 检查: PRD版本号 = Tech版本号（如 v1.0 对应 v1.0）

□ PRD变更记录与Tech变更记录一致
   └─ 检查: PRD版本更新记录 = Tech版本更新记录

□ Gate评审状态
   └─ 检查: PRD有CTO签字，Tech有PM+CTO双签

□ 代码实现范围与Tech文档一致
   └─ 检查: Task描述中的开发范围 ≤ Tech文档描述的范围

□ 相关文档已git push
   └─ 检查: PRD和Tech文档在远程仓库
```

### 5.2 一致性检查报告格式

```markdown
# 一致性检查报告

## 检查时间: {YYYY-MM-DD HH:mm}
## 检查人: Engineer Agent

### 版本对应检查
| 文档 | 版本 | 对应状态 |
|------|------|---------|
| PRD | v1.2 | ✅ 与Tech版本对应 |
| Tech | v1.2 | ✅ 与PRD版本对应 |

### Gate评审状态
| Gate | 状态 | 签字 | 日期 |
|------|------|------|------|
| Gate 1 (CTO技术评审) | ✅ 通过 | CTO: 2026-03-26 | |
| Gate 2 (Tech评审) | ✅ 通过 | PM+CTO: 2026-03-26 | |

### 开发范围确认
□ 开发范围: [功能A, 功能B]
□ Tech覆盖: [功能A, 功能B, 功能C]
□ 范围状态: ✅ 一致

### 结论
✅ **一致性检查通过**，可以开始实施

【⚠️ 如有不一致】
❌ **一致性检查不通过**
□ PRD版本 [v1.1] 与 Tech版本 [v1.0] 不对应
□ 建议: 等待PM完成Tech同步后再开始实施
```

### 5.3 一致性检查触发时机

| 时机 | 是否需要检查 |
|------|------------|
| 开始新任务前 | 必须 |
| 任务中断后恢复 | 必须 |
| 每日DBR #1前 | 可选 |

## 六、PM任务指派规范

### 6.1 变更任务指派流程

```
变更评审通过后:
     ↓
PM创建Task到TaskList
     ↓
Engineer从TaskList领取任务
     ↓
Engineer执行一致性检查
     ↓
├── ✅通过 → 开始实施
└── ❌不通过 → 返回PM处理

禁止路径:
❌ PM直接通过消息告诉Engineer变更需求
❌ PM直接指派Engineer执行未经评审的变更
```

### 6.2 PM指派权限约束

| 场景 | PM权限 | 说明 |
|------|--------|------|
| 小变更 (S) | 可直接指派 | 无需Gate评审 |
| 中变更 (M) | CTO局部评审后指派 | CTO确认Tech更新后可指派 |
| 大变更 (L) | 完整Gate评审后指派 | 必须Gate 1+Gate 2通过后才能指派 |

### 6.3 违规处理

```
PM违规处理:
| 违规类型 | 处理方式 |
|---------|---------|
| 直接告诉Engineer变更需求（未经评审） | 记录违规，PM+CTO对齐 |
| 跳过Gate评审直接指派大变更 | CTO拒绝，PM重新发起正式评审 |
| 连续2次违规 | Team Lead介入 |

Engineer违规处理:
| 违规类型 | 处理方式 |
|---------|---------|
| 未做一致性检查直接实施 | 回滚代码，重新检查 |
| 接收未经Gate评审的任务 | Engineer拒绝，PM重新发起评审 |
| 连续2次违规 | Team Lead介入 |
```

---

## 七、禁止规则汇总

- ❌ PM不得跳过Gate评审直接指派Engineer
- ❌ Engineer不得接收未经Gate评审的变更任务
- ❌ 一致性检查不通过不得开始实施
- ❌ 评审不通过不得继续下一步

## 八、版本号对应规则

| PRD版本 | Tech版本 | 说明 |
|---------|---------|------|
| v1.0 | v1.0 | 初始版本，完全对应 |
| v1.1 | v1.0 | 小变更，Tech无需同步 |
| v1.2 | v1.2 | 中变更，Tech局部同步后对应 |
| v2.0 | v2.0 | 大变更，完整Gate评审后全面同步 |

**对应原则**: PRD版本号与Tech版本号在变更完成后必须一致（S变更除外）。
