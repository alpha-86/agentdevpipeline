# 044 Phase 5.1 依赖盘点

## 结论

当前 `agentdevpipeline` 的依赖形态，不是“语言运行时依赖很多”，而是“研发协作依赖需要单独收口”。

本轮盘点后，确认如下：

1. 仓库当前没有复杂的应用运行时依赖

- 无 `package.json`
- 无 `pyproject.toml`
- 无 `requirements.txt`
- 无 `go.mod`

2. 当前唯一正式插件元数据入口是：

- `plugins/agentdevpipeline/.codex-plugin/plugin.json`

3. 当前真正需要说明的依赖是协作环境依赖

- `git`
- `rg`
- Markdown 文件系统工作流
- Issue 跟踪系统
- Pull Request 工作流
- `gh`
- stacked PR 工具（`git-stack` / `gstack` / 等价方案）
- CI 平台

4. 外部能力的定位需要继续分层

- `superpowers`：适合参考目录结构、技能拆分和执行方式
- `gstack`：适合作为 stacked PR、评审、发布、检查类能力的宿主环境
- GitHub CLI / GitHub 平台能力：是当前最贴近插件主流程的外部协作依赖
