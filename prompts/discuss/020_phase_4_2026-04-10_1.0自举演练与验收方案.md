# 020 Phase 4 - 1.0 自举演练与验收方案

## 目的

本文件用于执行 [020_2026-04-10_AgentDevFlow_1.0_自举里程碑执行计划](/home/work/code/AgentDevFlow/prompts/discuss/020_2026-04-10_AgentDevFlow_1.0_自举里程碑执行计划.md) 的 `Phase 4`，把 1.0 自举从“原则”和“平台方案”进一步收成可执行演练。

本阶段回答 5 个问题：

1. 用 AgentDevFlow 自举 AgentDevFlow 时，具体做哪个真实任务
2. Claude 侧如何跑
3. Codex 侧如何跑
4. 通过 / 失败如何判定
5. 需要留下哪些证据

## 结论摘要

本轮 1.0 自举演练选择的真实任务是：

**把 AgentDevFlow 自己的对外入口和平台接入文档，从“迁移/接入说明”升级为“真正可自举的运行入口”。**

原因：

1. 该任务直接服务 1.0 目标，不是虚构 demo
2. 它天然覆盖：
   - 独立性口径
   - 角色体系修正
   - Claude / Codex 平台入口
   - 复用原则
   - 启动顺序与 Gate
3. 它足以验证 AgentDevFlow 是否真能迭代自己，而不是只会描述别人

## 一、演练任务定义

## 任务名称

`AgentDevFlow 平台自举入口收口`

## 任务目标

完成以下真实改动：

1. 重写根 `README.md`
   - 不再以 `hedge-ai` 作为对外理解前提
   - 直接说明 AgentDevFlow 解决什么问题、如何启动、如何自举
2. 重写 `plugins/agentdevflow/README.md`
   - 变成 Codex 自举入口，而不是一般说明
3. 重写 `docs/platforms/claude-code.md`
   - 变成 Claude 自举入口，而不是最小安装说明
4. 重写 `docs/platforms/codex.md`
   - 变成 Codex 自举入口，而不是最小安装说明
5. 修正 `adapters/claude/.claude/skills/TEAM_SETUP.md`
   - 成为 Claude 真实入口
   - 去掉失效路径 `prompts/zh-cn/`
6. 视需要补充：
   - 角色命名与默认团队说明
   - 必读链路
   - 自举启动步骤
   - 最小检查与失败阻断规则

## 为什么选这个任务

1. 它同时检验文档阶段、平台阶段和多角色协作
2. 它要求 AgentDevFlow 对自己做产品、方案、实现和验证
3. 它能直接为 `Phase 5 / Phase 6` 提供最终可用入口

## 二、共同验收范围

无论 Claude 还是 Codex，演练必须覆盖下列能力：

1. 需求澄清
2. PRD / 方案明确
3. 架构 / 平台方案收敛
4. QA / 验证视角介入
5. 文档阶段确认
6. 具体改动执行
7. 验证与收口

## 产物范围

至少应产生：

1. 一份启动会或主线纪要
2. 一份主 issue / 主工作索引
3. 文档阶段结论
4. 实现阶段结论
5. 验证结论
6. 演练报告

## 三、Claude 演练脚本

## 前置条件

1. Claude 可读取本仓库
2. Claude 可使用本地 skill / Agent Team 能力
3. Team Lead 可先读取 AgentDevFlow 共享入口

## Claude 演练步骤

### 步骤 0：装载入口

1. 读取：
   - `adapters/claude/.claude/skills/TEAM_SETUP.md`
   - `skills/shared/start-agent-team.md`
   - `skills/shared/create-agent.md`
   - `docs/governance/core-principles.md`
2. 明确本轮主任务为：
   - “收口 AgentDevFlow 自身的 Claude / Codex 自举入口”

### 步骤 1：启动团队

按顺序创建：

1. Team Lead
2. Product Manager
3. 架构师
4. QA Engineer
5. Engineer
6. Process Auditor

按需：

1. Platform/SRE
2. 研究支持

### 步骤 2：文档阶段

1. PM 澄清任务边界
2. 架构师给出平台入口与共享入口方案
3. QA 给出文档和接入验证点
4. Team Lead 组织文档阶段确认
5. 形成人工评审 #1 结论

### 步骤 3：实现阶段

1. Engineer 修改目标文件
2. QA 做最小验证
3. Team Lead 确认是否进入人工评审 #2

### 步骤 4：收口

1. 输出验证结论
2. 回写主 issue / 纪要 / 记录
3. 形成 Claude 演练报告

## Claude 通过标准

1. 团队启动顺序正确
2. Team Lead 先读共享入口而不是直接开工
3. 文档阶段和实现阶段分离
4. 人审节点存在且可追溯
5. 最终改动真实落到仓库中

## 四、Codex 演练脚本

## 前置条件

1. Codex 可访问本仓库
2. Codex 可使用多 agent 调度能力
3. 主会话承担 Team Lead 等效编排职责

## Codex 演练步骤

### 步骤 0：编排者初始化

主会话先读取：

1. `plugins/agentdevflow/README.md`
2. `skills/shared/start-agent-team.md`
3. `skills/shared/create-agent.md`
4. `docs/governance/core-principles.md`
5. `020_phase_3_2026-04-10_Claude与Codex对等自举方案.md`

然后明确：

1. project_id
2. 主 issue / 主工作索引
3. 当前 Gate
4. 需要创建的角色 subagent

### 步骤 1：创建文档阶段角色

主会话创建：

1. Product Manager subagent
2. 架构师 subagent
3. QA Engineer subagent

每个 subagent 都必须收到：

1. 角色定义
2. 必读文档
3. 当前任务边界
4. 允许写入范围
5. 初始化确认要求

### 步骤 2：整合文档阶段结论

主会话负责：

1. 汇总 PM / 架构师 / QA 结果
2. 判断是否进入文档确认
3. 形成人工评审 #1 结论

### 步骤 3：创建实现阶段角色

在文档阶段通过后，主会话创建：

1. Engineer subagent
2. Process Auditor subagent

按需：

1. Platform/SRE subagent

### 步骤 4：实现与验证

1. Engineer 完成文件改动
2. QA 或 Process Auditor 验证入口、路径、角色和阻断规则
3. 主会话形成人工评审 #2 结论

### 步骤 5：收口

1. 输出 Codex 演练报告
2. 回写主 issue / 纪要 / 记录
3. 关闭或切换当前 Gate

## Codex 通过标准

1. 主会话确实承担 Team Lead 等效编排职责
2. 各角色 subagent 具备明确职责和写入边界
3. 文档阶段先于实现阶段
4. 主会话统一把守 Gate，而不是任由 subagent 各自推进
5. 最终改动真实落到仓库中

## 五、统一证据要求

每次演练至少保留以下证据：

1. 启动记录
   - 本轮 project_id
   - 主 issue
   - 启用角色
   - 当前 Gate
2. 文档阶段结论
   - 需求边界
   - 平台方案
   - QA 验证点
   - 文档确认结论
3. 实现阶段结论
   - 修改文件
   - 验证方式
   - 风险与残留项
4. 最终演练报告
   - 是否通过
   - 失败点
   - 回退建议

## 推荐落点

推荐把演练证据落到：

1. `prompts/discuss/`
2. `docs/memo/`
3. `docs/qa/`

## 六、失败标准

以下任一出现，演练视为失败：

1. 角色未正确装载就开始正式工作
2. 文档阶段和实现阶段混在一起
3. 人审节点缺失
4. Comment / 记录未真实落地
5. 关键文件虽然被修改，但无法解释为什么这样改
6. Codex 侧没有体现真实多 agent 编排
7. Claude 侧没有体现真实 Agent Team 约束

## 七、回退要求

若演练失败，必须至少输出：

1. 失败发生在哪个阶段
2. 缺的是规则、入口、角色约束，还是平台能力
3. 是否可在当前 phase 修补
4. 若不能，必须回退到：
   - `Phase 2` 边界修正
   - 或 `Phase 3` 平台方案修正

## 八、Phase 5 / 6 的执行顺序

建议顺序：

1. 先执行 Claude 演练
2. 修补 Claude 侧暴露的问题
3. 再执行 Codex 演练
4. 基于 Codex 结果再做统一收口

原因：

- Claude 侧已有 Team 宿主，更适合作为第一轮验证
- Codex 侧编排更灵活，也更容易暴露“规则有没有真的被装进多 agent”这个问题

## 九、Phase 4 最终判断

本阶段已经把 1.0 自举从“概念目标”收成了可执行演练。

下一步不再讨论“拿什么任务做验证”，而是直接进入：

1. `Phase 5` Claude 自举演练
2. `Phase 6` Codex 自举演练

并按每个 phase 独立提交、独立 push 的要求推进。
