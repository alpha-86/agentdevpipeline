# 015 事件总线到平台自动化映射与 Gate Queue 差距审计

## 审计背景

上一轮已经补了共享事件总线、团队启动入口和 Agent 创建入口，但仍有一个明显缺口：事件模型和平台适配层之间还缺一层更接近运行态的桥梁。

这会导致：

- event bus 仍像协议文档，不像可接入自动化的资产
- 各平台只知道要做最小检查，但不知道事件驱动下如何落地
- 关键 Gate 没有一个统一的排队视图来观察阻断、缺项和下一动作

## 本轮重点

### 1. 新增运行态模板

- `skills/shared/templates/event-record-template.md`
- `skills/shared/templates/gate-queue-template.md`

### 2. 新增运行态示例

- `docs/memo/event_bus_trace_2026-04-09_example.md`
- `docs/memo/gate_queue_2026-04-09_example.md`

### 3. 新增平台映射样例

- `adapters/claude/event-automation-example.md`
- `adapters/codex/event-automation-example.md`
- `adapters/opencode/event-automation-example.md`

## 审计结论

这轮的重点不是再写原则，而是把共享事件模型和平台接入之间的中间层补出来。这样项目从“有规则”进一步走向“有运行态骨架”。

仍然存在的缺口：

1. 还没有真正的脚本或 Action 实现
2. Gate Queue 还是示例，不是实时数据源
3. event record 还没有标准化输出到统一目录或数据库
