## 四、Team Lead 唯一性 / 创建契约专项分析

### 4.1 源文件核心规则（基于 `019_TeamLead职责边界.md` 和 `create-agent/SKILL.md`）

**Team Lead 禁止行为（关键）：**
1. 写任何代码（Python/shell/SQL/配置文件修改）—— 绝对红线
2. 直接修改 PRD/Tech/Strategy 文档（除会议记录外）—— 绝对红线
3. 执行 Bash 命令（git/diff/log 查看除外）—— 绝对红线
4. 跳过评审流程直接做决策
5. 代替其他 Agent 做本应属于他们职责的工作
6. **trigger：发消息让 Agent"请开始执行"** —— 违反 AI驱动AI
7. **不作为：发现 pending 任务但不分配** —— Team Lead 核心职责是任务分配

**创建契约（关键）：**
- 创建任何 Agent 前必须使用 `create-agent` skill
- 必须先确保 Team "hedge-ai" 存在
- team-lead 类型 → 返回提示"Team Lead 是你本身，不需要创建"
- 必须读取 skill 文件并提取 Critical Rules、核心职责、必读文档等
- 创建后向 Human 确认

**"分配≠trigger执行"区分（V2.1 核心）：**
- **分配**：告知责任归属，用 TaskUpdate 设 owner
- **trigger**：命令执行，违反 AI驱动AI 原则
- 正确做法："Task #X 已分配给你，请领取并执行"
- 错误做法："请开始执行 Task #X"

## 五、Gate 流程保障专项分析

### 5.1 源文件核心机制（基于 `V2.3_Gate_流程保障机制.md` 和 `V2.2.5_Issue_Comment机制修复与强制Gate建立.md`）

**Gate 定义（关键）：**

| Gate | 文档类型 | 必需签字 | 检查位置 |
|------|---------|---------|----------|
| **PRD Gate 1** | PRD | PM + 架构评审人 + 质量评审人 | PRD 文档头部 Gate 块 |
| **Tech Gate 2** | Tech | PM + 技术评审人 + QA | Tech 文档头部 Gate 块 |
| **QA Case Design** | QA Case | PM + CTO + Engineer (三方签字) | QA Case 文档头部 Gate 块 |
| **QA Test Report** | QA Test Report | CTO + Engineer + PM (三方签字验收) | QA Test Report 文档头部 Gate 块 |

**三层保障体系（V2.3 核心）：**

1. **第一层：GitHub 原生能力**
   - Branch Protection Rules → 强制 PR 才能合并到 main
   - PR Template → 强制信息完整
   - Merge Gate → 条件满足才能合并

2. **第二层：CI/CD 自动检查**
   - `doc-pr-checks.yml` → 文档 PR 完整性检查 + Gate 状态检查
   - `code-pr-checks.yml` → 代码 PR 前置条件检查
   - `issue-status-sync.yml` → Issue 状态同步检查

3. **第三层：Agent Critical Rules**
   - PM: 禁止跳过文档 PR 指派开发
   - Engineer: 开发前必须确认文档 PR 已合并
   - QA: Case Design 必须前置到文档 PR

**强制 Gate 阻断机制（V2.2.5 关键）：**
- 未 Comment 到 Issue 阻塞 PR 创建
- Issue 下至少有 1 条 Agent Comment
- Comment 内容包含产物链接（PRD/Tech/PR）
- V2.2.4: 未 Comment 视为未完成

**友好错误信息（V2.3 新增）：**
```
[Gate Check] ❌ 前置条件未满足
问题: 文档 PR 提交时，关联的 Tech Gate 2 尚未全部签字
要求:
  □ PM 签字: ❌ 未通过
  □ CTO 签字: ✅ 已通过
解决方案:
  1. 完成所有 Gate 签字后再提交 PR
```

## 六、PRD/Tech/代码一致性专项分析

### 6.1 源文件核心机制（基于 `V1.0_PRD_Tech_代码一致性变更管理规范.md`）

**变更等级定义（关键）：**

| 等级 | 定义 | Tech同步 | Gate评审 | 版本号规则 |
|------|------|---------|---------|-----------|
| **S (小变更)** | 仅修改PRD文字描述、格式调整、错别字修正 | 无需 | 无需 | v1.0 → v1.1 |
| **M (中变更)** | 修改功能细节、验收标准调整、优先级调整 | 局部同步 | 局部评审 | v1.0 → v1.2 |
| **L (大变更)** | 影响系统架构、模块边界、数据模型、外部接口 | 全面同步 | 完整Gate评审 | v1.0 → v2.0 |

**Engineer 一致性检查（强制）：**

```
Engineer实施前一致性检查清单 V1.0

□ PRD版本与Tech版本一致
   └─ 检查: PRD版本号 = Tech版本号

□ PRD变更记录与Tech变更记录一致
   └─ 检查: PRD版本更新记录 = Tech版本更新记录

□ Gate评审状态
   └─ 检查: PRD有CTO签字，Tech有PM+CTO双签

□ 代码实现范围与Tech文档一致
   └─ 检查: Task描述中的开发范围 ≤ Tech文档描述的范围

□ 相关文档已git push
   └─ 检查: PRD和Tech文档在远程仓库
```

**一致性检查报告格式（V1.0 要求）：**
- 版本对应检查（PRD v1.x ↔ Tech v1.x）
- Gate 评审状态（CTO 签字日期等）
- 开发范围确认
- 不通过时的建议处理

**PM 指派权限约束：**

| 场景 | PM权限 | 说明 |
|------|--------|------|
| 小变更 (S) | 可直接指派 | 无需 Gate 评审 |
| 中变更 (M) | CTO局部评审后指派 | CTO 确认 Tech 更新后可指派 |
| 大变更 (L) | 完整 Gate 评审后指派 | 必须 Gate 1+Gate 2 通过后才能指派 |

**禁止规则：**
- PM 不得跳过 Gate 评审直接指派 Engineer
- Engineer 不得接收未经 Gate 评审的变更任务
- 一致性检查不通过不得开始实施
