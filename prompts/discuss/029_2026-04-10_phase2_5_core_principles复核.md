# 029 - 2026-04-10 Phase 2.5 core-principles 复核

## 目标

确认 `docs/zh-cn/governance/core-principles.md` 是否严格来自 `hedge-ai` 源文档抽取，而不是基于个人理解自由概括。

## 当前项目原有问题

旧版 `core-principles.md` 的主要问题不是“方向错”，而是“抽象过度”：

- 原则数量偏多
- 有些原则虽然合理，但源文档支撑不够直接
- 更像我自己的治理总结，而不是从 `hedge-ai` 迁出来的顶层方法

这不符合当前任务目标。

## 本次调整

### 1. 改成“每条原则都标来源”

新版 `core-principles.md` 不再只写原则标题，而是给每条原则补上明确来源，例如：

- `V2.0` 对应 `Issue First`、`问题先于方案`
- `V2.2` 对应 `Human Review`、`双阶段交付`
- `V2.2.5` 对应 `正式留痕优先`、`Gate 不可绕过`
- `V2.6` 对应 `合规检查是独立治理职责`

### 2. 删掉抽象过度的原则

旧版里这类表述虽然不一定错，但源支撑不够直接，因此不再保留为核心原则主干：

- “恢复优先于继续执行”
- “平台适配不能改写核心语义”

这些内容后续如有需要，可放在协议或实现约束里，但不应继续作为顶层核心原则。

### 3. 保留迁移边界原则

“只迁移通用研发机制”这一条保留，因为这是本项目相对 `hedge-ai` 的硬边界，且已经通过全量源文档审计固定下来。

## 结论

`core-principles.md` 现在更接近“源文档抽取结果”，不再是我自己的方法论扩展版本。

当前保留下来的核心原则主干是：

1. Issue First
2. 问题先于方案
3. Human Review 必须显式存在
4. 双阶段交付
5. 正式留痕优先
6. Gate 不可绕过
7. 合规检查是独立治理职责
8. 只迁移通用研发机制

下一步进入 `Phase 2.6`，只处理 README 的完整流程图与主线表达是否准确。
