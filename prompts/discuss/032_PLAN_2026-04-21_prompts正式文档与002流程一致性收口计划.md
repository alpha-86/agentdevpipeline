## 三、已确认的关键口径

Claude 必须严格按以下口径收口：

### 1. 流程图边界

- `流程图 A` = 单个 Issue 的正向交付流程
- `流程图 B` = 需求变更与打回流程
- `流程图 C` = Agent Team 启动后的 Issue 扫描与路由
- 不允许再把 `流程图 C` 的前缀步骤混入 `流程图 A`

### 2. Human Review 边界

- `Human Review #1` = 文档 PR 的 Human review / merge decision
- `Human Review #2` = 代码 PR 的 Human review / merge decision
- `HR#1` / `HR#2` 不是 Agent 签字节点
- Agent 可以在进入 HR 前完成自己的评审、签字、报告与结论
- 但 `HR#1` / `HR#2` 本身不等于 Agent 三方签字

### 3. 纯文档交付语义

- 在 AgentDevFlow 项目语义下，纯文档交付与开发交付同等重要
- 很多正式交付物本身就是 prompts、spec、workflow、review 记录等 Markdown 文档
- 纯文档交付不是例外，也不能弱化流程要求
- 纯文档交付同样必须严格经过：
  - GitHub Issue 主线
  - 正式文档产出
  - 文档 PR
  - Human Review #1
  - Issue Comment 回写
  - Human 执行 Issue 关闭

### 4. 实现启动边界

- `文档 PR 合并 = 设计确认`
- 文档 PR 合并后，Engineer 可进入 Implementation
- 不再把“Human 通知 Engineer 开始实现”写成必要前置条件

### 5. 权限边界

- PR 合并是 Human 专属操作
- Issue 关闭是 Human 专属操作
- Agent 只能发布关闭请求评论，不能替代 Human 执行关闭
- 不允许出现“通知 Team Lead 代为关闭”这类破坏边界的表述

### 6. 当前 `002` 口径下的关键签字矩阵

- Gate 1: `PM + Architect + QA`
- Gate 2: `QA + Engineer + PM`
- QA Case Design: `PM + Architect + Engineer`
- QA 测试报告评审: `PM + Architect + Engineer`
- Gate 5 Release 放行: `PM + Architect + Platform/SRE`
