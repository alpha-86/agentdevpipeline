## 一、总结论

用户指出的问题成立。

当前 AgentDevFlow 在 `prompts/zh-cn/*` 层已经开始沉淀规则，但 `skills/shared/agents/*` 仍明显偏薄，主要问题不是“缺少角色名称”，而是“缺少可执行性”：

- 更像角色简介，不像操作手册
- 有职责，但缺初始化动作
- 有规则，但缺会前必读和工作前检查
- 有阶段概念，但缺对当前 workflow 的强绑定
- 有 issue / human review 原则，但缺到角色级别的明确责任
- 缺上下文恢复步骤
- 缺输出格式和拒绝/退回格式
- 缺明确禁止行为
- 缺独立流程治理角色

这也是为什么当前 `agents` 给人的感觉不像从 `hedge-ai` 迁移来的，而更像是“基于理解重新写了一版摘要”。

## 二、从 hedge-ai 抽出来的可迁移结构要求

`hedge-ai` 的可迁移价值不在业务角色，而在每个 agent 定义的结构完整度。其共享特征包括：

1. 角色文件不是单纯的“职责说明”，而是“执行规范”。
2. 每个角色都有明确的 `Critical Rules`。
3. 每个角色都有成体系的 `Required Reading`。
4. 每个角色都有 `初始化后必做`。
5. 会议或评审前必须读取对应 `WORKFLOW`。
6. 会话清空后有明确 `上下文恢复` 路径。
7. Issue / Comment / Gate 责任会下沉到具体角色。
8. 明确列出 `禁止行为`，而不是只写“应该做什么”。
9. 对关键交付有固定输出格式，而不是自由发挥。
10. 存在独立的流程治理/合规检查角色，而不是完全依赖 Team Lead 顺手兼任。

## 三、当前 AgentDevFlow 的 P0 差距

### P0-1. `agents/*` 过薄

当前多数角色只有这些段落：

- 角色定义
- Critical Rules
- Responsibilities
- Standard Actions

这远不足以支撑多 agent、跨阶段、可恢复的真实运行。

### P0-2. 必读文档机制不完整

虽然已有 `006_agent_creation_contract.md`，但共享 agent 文件本身没有把“角色文件 + prompts + workflows + templates”的强制阅读关系写清楚。

### P0-3. 初始化与上下文恢复缺失

`hedge-ai` 的 agent 在启动和压缩恢复后都有明确动作，当前项目大多没有。

### P0-4. 缺少独立流程治理角色

当前只有 Team Lead 承担监督责任，但对“谁主动发现流程漂移、Issue 漏链、Gate 缺签、Todo 缺 owner”没有一个稳定角色。

`hedge-ai` 的 `PMO-Agent` 在这里的通用抽象是需要保留的。

## 四、逐角色对照结论

### 4.1 Team Lead

`hedge-ai` 优点：

- 有完整必读文档清单
- 有会议前强制读取 workflow 的要求
- 有每日工作循环
- 有评审升级机制
- 有上下文恢复和输出格式

当前缺口：

- 只有职责和几条规则
- 没有会议前检查和 workflow 映射
- 没有输出格式
- 没有恢复路径

本轮改进：

- 补初始化
- 补会议前准备
- 补日常工作循环
- 补上下文恢复
- 补输出格式
- 补禁止行为

### 4.2 Product Manager

`hedge-ai` 优点：

- 明确要求 Issue First
- 强调讨论问题而不是直接接受方案
- PRD 生命周期完整
- 文档阶段和代码阶段边界清晰
- Comment 节点明确

当前缺口：

- PRD 职责有了，但没有完整生命周期
- 对 Human Review #1 和文档阶段完整交付的责任不够明确
- 没有 issue 讨论、评论、恢复和拒绝越权的具体动作

本轮改进：

- 强化 Issue 驱动
- 强化文档阶段完整性交付
- 强化 Human Review #1 责任
- 补 PRD 生命周期、上下文恢复、输出格式、禁止行为

### 4.3 Tech Lead

`hedge-ai` 优点：

- 技术评审职责非常完整
- 有 Tech 文档位置规则
- 有 Issue 评论格式
- 有初始化和待评审检查

当前缺口：

- 只有“写 Tech Spec 和做评审”的概述
- 没有初始化、评审前检查、Issue 评论责任、输出格式

本轮改进：

- 补评审检查清单
- 补文档阶段责任
- 补上下文恢复和 issue comment 责任
- 补禁止行为和输出格式

### 4.4 QA Engineer

`hedge-ai` 优点：

- 明确持有 Tech Review Gate
- Case 设计、测试报告、Issue 关联都写得很完整
- 有初始化和自动执行机制

当前缺口：

- 只有“设计 case、执行验证、出报告”的摘要
- 还不够像一个真正的 gate owner

本轮改进：

- 明确 QA 在 Tech Review 和 Release 决策中的 gate 责任
- 强化 QA Case Design 与测试报告的文档阶段位置
- 补 issue comment、恢复、输出格式和禁止行为

### 4.5 Engineer

`hedge-ai` 优点：

- 强调开发前置条件和开发拒绝
- 明确文档阶段确认完成后才能编码
- 明确 PR / Issue 关联和交接路径

当前缺口：

- 对“何时拒绝开始开发”写得不够强
- 缺少实现阶段与文档阶段边界的强约束
- 缺 issue comment / 交接格式

本轮改进：

- 强化开发前置条件
- 补开发拒绝逻辑
- 补 QA 交接、issue comment、恢复和输出格式

### 4.6 Platform/SRE

`hedge-ai` 的 `sre-agent.md` 业务耦合很强，但可迁移的结构仍包括：

- 初始化
- 运行风险阻断
- 发布与回滚证据
- 异常升级

当前缺口：

- 只有职责摘要
- 与平台检查层、发布证据、异常升级的绑定太弱

本轮改进：

- 强化与 `platform-minimum-checks` 的连接
- 补初始化、上下文恢复、发布前检查、输出格式

### 4.7 Researcher

`hedge-ai` 的 `research-agent.md` 虽然业务语义不可迁移，但结构上有两点值得保留：

- 明确问题定义和证据要求
- 明确输出格式与上下文恢复

当前缺口：

- Researcher 仍像一个泛用说明，没有形成“带约束的探索角色”

本轮改进：

- 强化问题定义、假设、风险、建议输出
- 补初始化、恢复、禁止行为、输出格式

### 4.8 PMO-Agent

这是当前项目最明显的角色空洞。

`hedge-ai` 的 `PMO-Agent` 虽然实现还较轻，但它代表了一个关键思想：流程合规检查不能完全依赖 Team Lead 临时兼任。

本项目的通用化改造方向应为：

- 新增 `Process Auditor`
- 职责是审计 Gate、Issue、Comment、Todo、例外审批和状态漂移
- 不取代 PM / Tech Lead / QA 的签字责任
- 但负责主动发现违规并触发纠正

## 五、对 V3.0 规则层的补充结论

除 `AGENTS` 外，`hedge-ai` 还有三条被本项目此前吸收不彻底的结构要求：

### 5.1 Team / Agent 启动顺序

`TEAM_SETUP.md` 明确要求：

- 先建立团队
- 再按顺序创建角色
- 每个角色创建后必须先读角色文件和相关文档

本项目当前缺口：

- `006_agent_creation_contract.md` 还不够具体

本轮改进方向：

- 把“角色文件 + prompts + workflows + templates + 初始化检查”写进创建契约

### 5.2 Skill / Workflow 强绑定

`hedge-ai` 明确把 workflow 当成会议和评审的前置阅读物，而不是附录。

本项目当前缺口：

- workflow 虽有，但 agent 文件对 workflow 的依赖关系偏弱

本轮改进方向：

- 在每个角色文件中补“会前/评审前必须读取的 workflow”

### 5.3 输出格式显式化

`hedge-ai` 大量使用固定输出格式，这对多 agent 协作和上下文恢复非常重要。

本项目当前缺口：

- 只有模板层，没有在角色层规定“哪些结论必须按何种结构输出”

本轮改进方向：

- 在角色文件中补简化版输出格式

## 七、仍未完成的后续项

即使本轮角色文件补厚后，距离 `hedge-ai` 那种“相对完整、可执行、可实时”的状态，仍有剩余工作：

- 还需要把 `start-agent-team` / `create-agent` 类的启动机制在本项目里系统化
- 还需要继续把 workflow 与模板的字段做强绑定
- 还需要补平台检查脚本或适配层示例
- 还需要把英文版整体镜像到位

## 八、结论

本轮最重要的不是“再加一个新概念”，而是把 agent 层从“说明书”升级成“执行手册”。

只有角色文件足够具体，`Issue First`、`Human In The Loop`、`双阶段 PR`、`三层保障体系` 这些原则才能真正落到执行层，而不只是留在治理文档里。
