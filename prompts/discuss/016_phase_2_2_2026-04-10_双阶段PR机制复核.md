# 026 - 2026-04-10 Phase 2.2 双阶段 PR 机制复核

## 对照源文件

- `hedge-ai/prompts/V3.0/change_record/V2.2_双PR机制引入.md`

## 本次只复核什么

本次只复核这三块：

1. 双阶段 PR 机制
2. QA Case Design 前置到文档 PR
3. 三层保障体系

本次不处理：

- Issue Comment Gate 的细化
- PMO 合规主动检查
- README 流程图最终改写

## 当前项目原有问题

当前仓库虽然已经有：

- `017_human_review_and_signoff.md`
- `019_dual_stage_pr_and_three_layer_safeguard.md`
- `skills/shared/workflows/human-review.md`

但表达还偏抽象，离 `hedge-ai V2.2` 的具体约束还有差距，主要缺：

- 没把文档 PR 与代码 PR 的分支和前置关系写清
- 没把 `PRD + Tech + QA Case Design` 明确写成文档 PR 的最小必需集
- 没把代码 PR 必须附文档 PR 链接和测试报告写清
- 三层保障体系还不够接近源文档的“平台原生能力 / CI 检查 / Agent Critical Rules”分层

## 本次落地调整

### 1. 补齐双阶段 PR 的最小定义

明确写入：

- 文档阶段使用 `doc-{issue}-{描述}`
- 实现阶段使用 `feature-{issue}-{描述}`
- 文档 PR 合并 = 设计确认
- 代码 PR 合并 = 实现确认

### 2. 补齐文档 PR / 代码 PR 的最小内容

文档 PR 必须包含：

- PRD
- Tech
- QA Case Design
- 变更摘要

代码 PR 必须包含：

- 已合并的文档 PR 链接
- 测试报告
- 代码变更摘要
- 关联 Issue

### 3. 补齐三层保障体系的具体含义

现在已经明确成：

- 第一层：流程规则
- 第二层：执行留痕
- 第三层：平台与自动化检查

并把第三层进一步贴近源文档写成：

- Branch Protection / PR Template / Merge Gate
- 文档 PR 完整性检查
- 代码 PR 前置条件检查
- Issue 状态一致性检查
- Comment / artifact linkage 检查

## 结论

当前项目的双阶段 PR 机制，现在比之前更接近 `hedge-ai V2.2` 的原始定义了，尤其是：

- 不再只说“设计确认”和“实现确认”，而是明确落到文档 PR / 代码 PR
- QA Case Design 不再只是可选补充，而是文档 PR 的必需组成部分
- 三层保障体系不再只是口号，而是明确分层

但这一项还没有完全结束整个双阶段主线，剩余内容会在后续项继续推进：

- Issue Comment Gate
- README 中的完整流程图
- 更贴近源文档的插件入口和 CHANGELOG 机制
