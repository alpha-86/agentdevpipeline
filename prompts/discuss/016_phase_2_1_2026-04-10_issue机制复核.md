# 025 - 2026-04-10 Phase 2.1 Issue 机制复核

## 对照源文件

- `hedge-ai/prompts/V3.0/change_record/V2.0_研发流程与GitHubIssue结合及QA角色引入.md`

## 本次只复核什么

本次只复核 `Issue First` 主线，不把双阶段 PR、Issue Comment Gate、PMO 合规检查混在这一项里一起做。

重点只看这几条源机制：

1. 所有项目必须以 GitHub Issue 为入口
2. PM 与用户讨论时不能直接接受方案，要回归问题本身
3. 每次讨论结果必须 comment 到 Issue
4. PRD 只描述问题和解决方案，不含技术实现
5. PRD 完成后必须 comment 到 Issue
6. 所有产物最终都要关联到 Issue
7. Issue 关闭由 Human 完成

## 当前项目原有问题

原来的 `007_issue_driven_orchestration.md` 已经表达了 “无 Issue 不进入正式流程” 和 “Issue 是唯一主索引”，但还不够接近源文档，主要缺这几条硬约束：

- 缺 “PM 遇到用户给出方案时，要回归问题本身” 的明确规则
- 缺 “每轮正式讨论后必须 comment 到 Issue” 的明确规则
- 缺 “PRD 不写技术实现” 与 “PRD 完成后必须回写 Issue” 的明确规则
- 缺 “Issue 默认由 Human 最终关闭” 的明确规则

## 本次落地调整

### 1. 强化强制入口规则

新增了这些正式入口约束：

- 新需求、缺陷、异常、流程纠偏、发布后问题都必须先建 Issue
- 没有 Issue 时，禁止进入 PRD、Tech、开发、QA、发布放行

### 2. 补齐 PM 的 Issue 讨论规则

新增：

- 用户在 Issue 中给出方案时，PM 必须回到问题、目标、约束和边界
- 每轮正式讨论结束后都必须写 Issue Comment
- PRD 完成后必须把摘要和链接写回 Issue
- PRD 只写问题、目标、范围、产品方案和边界，不写技术实现

### 3. 补齐产物与关闭规则

新增：

- PRD / Tech / QA / Release / Todo / Memo / Change Record / Review Record 都必须按需回链到对应 Issue
- Issue 关闭前必须确认关键产物完整、当前 Gate 已完成、阻塞性缺陷已清除或获批例外
- Issue 默认不由 Agent 私自关闭，应由 Human 或明确授权的人类完成最终关闭

## 结论

`Issue First` 这一条，现在比之前更接近 `hedge-ai V2.0` 的原始要求了，尤其是：

- 从“推荐使用 Issue”收紧成“没有 Issue 不准开工”
- 从“Issue 留痕”收紧成“每轮正式讨论必须 comment”
- 从“流程追踪”收紧成“Human 最终关闭”

但这一步还没有覆盖 `V2.0` 里的全部内容。剩余仍需在后续项继续处理：

- QA 正式 Gate
- QA case 设计与验收签字
- 与双阶段 PR 的主流程整合
