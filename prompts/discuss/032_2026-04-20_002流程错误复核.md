## 一、变更原因

用户（Human）在 CEO Review 讨论中指出 `prompts/002_develop_pipeline.md` 存在**fundamental errors**，核心问题是：

1. **流程概览起点错误**：把"创建 GitHub Issue"作为起点，Team Startup 放在 Issue 之后。实际是 Agent Team 先启动，Team 启动后扫描所有 open Issue，Human 随时可能创建 Issue（不是启动前固定动作）。

2. **缺少 PM 领取 Issue + 讨论必须 Comment 规则**

3. **缺少双 PR 分支策略**

4. **缺少 V2.2.4 Issue Comment 强制要求**

参考文档：
- `hedge-ai/prompts/V3.0/change_record/V2.0_研发流程与GitHubIssue结合及QA角色引入.md`
- `hedge-ai/prompts/V3.0/change_record/V2.2_双PR机制引入.md`

## 二、完整错误清单

### 错误 1：流程概览起点错误（Lines 11-46）

**当前错误**：
```
需求 / 问题 / 异常
        │
        ▼
创建主 GitHub Issue
        │
        ▼
Gate 0: Team Startup
```

**问题**：暗示"先有 Issue，再启动 Team"。实际上 Agent Team 先启动，Team 启动后扫描所有 open Issue。Human 随时可能创建 Issue，不是启动前的固定动作。

**正确理解**：Team Startup 是初始化动作，不是 Issue 驱动的。Team 启动后 PM 从已扫描的 open Issues 中领取任务。

---

### 错误 2：Gate 0 触发条件错误（Line 64）

**当前**："新 Issue 创建后，Team Lead 分配角色"

**问题**：Team Startup 不依赖 Issue 创建。角色分配在 Team 初始化时完成，不是 Issue 创建触发的。

---

### 错误 3：缺少 PM 领取 Issue + 讨论必须 Comment 规则

**缺失内容**（V2.0 Section 2.2 Critical Rules）：
- PM 领取 Issue 后必须与 Human 讨论问题
- 每次讨论结果必须 Comment 到 Issue
- Issue 含方案时，PM 必须回归问题本身讨论
- **未 Comment 视为未完成**（V2.2.4）

---

### 错误 4：Gate 1 签字矩阵部分正确，但缺少"讨论 Comment"前置要求

**当前**：Gate 1 进入条件只要求"PRD 文档已起草"

**缺失**：
- PM 领取 Issue 后与 Human 讨论（必须 Comment）才能进入 PRD
- V2.0 明确："无 Issue 不进入 PRD"

---

### 错误 5：Gate 2 QA Case Design 角色描述与 V2.2 不完全一致

**当前**：Gate 2 进入条件已包含 QA Case Design（✅），签字要求也有 QA（✅）

**但**：
- V2.2 中 QA Case Design 是**文档 PR 的必需组成部分**
- 签字顺序：CTO Tech Review（PM + QA 两签）→ QA Case Design（三方：PM + CTO + Engineer）
- 当前描述把 QA Case Design 混在 Gate 2 内，未区分 CTO Review 和 Case Design 是两个不同阶段

---

### 错误 6：缺少双 PR 分支策略（V2.2 核心变更）

**缺失**：
```
doc-{issue_number}-{简短描述}  →  文档 PR（PRD + Tech + QA Case Design）
feature-{issue_number}-{简短描述}  →  代码 PR（代码 + 测试报告）
```

当前只说"文档 PR 合并"、"代码 PR 合并"，没有：
- 分支命名规范
- 生命周期（创建 → Review → 合并/关闭）
- 依赖关系（代码 PR 必须基于文档 PR 已合并）

---

### 错误 7：缺少 V2.2.4 Issue Comment 强制要求

**V2.2.4 明确规定**：
- 每个角色完成工作后必须在 Issue 下 Comment
- 每个交付物完成后必须 Comment 并附链接
- **未 Comment 视为未完成**

当前只在"输出物"中简单提了 Issue Comment，没有作为强制规则强调。

---

### 错误 8：Human Review #1 触发描述不够准确

**当前**："通过 → PM 发起 Human Review #1"

**问题**：
- V2.2 描述的是：PM 创建 `doc-{issue}` 分支 → 提交文档 → 创建 PR → Human Review
- Human Review 的触发是 PR 创建，不是 PM "发起"

---

### 错误 9：HR#1 和 HR#2 缺少与 PR 的绑定关系

**缺失**：
- Human Review #1 = 文档 PR 合并 = 设计确认
- Human Review #2 = 代码 PR 合并 = 实现确认

当前把 Human Review #1/#2 描述为独立节点，没有明确它们对应哪个 PR 的哪个阶段。

---

### 错误 10：PM 关闭 Issue 的具体方式描述不够准确

**当前**："Agent 只负责发布关闭请求评论"

**V2.2 明确**：
- PM 执行：`python scripts/github_issue_sync.py --close-issue --issue N --reason completed --close-comment "..."`
- Agent 发布"关闭请求"评论，Human 执行实际关闭

---

### 错误 11：流程概览缺少双 PR 阶段可视化

**当前概览**：
```
Human Review #1 — 设计确认评审
        │
        ▼
文档 PR 合并 = 设计确认 ✅
```

**问题**：没有体现文档 PR 和代码 PR 是两个独立的分支生命周期。

## 三、正确流程（基于 V2.0/V2.2）

### 3.1 正确流程概览

分为两个独立流程图：

#### 流程图 A：完整 Issue 交付流程（正向流程，从认领到关闭）

```
PM 领取 Issue → 与 Human 讨论问题（必须 Comment 到 Issue）
        │
        ▼
Gate 1: PRD Review — 需求评审（PM + Architect + QA 三方签字）
        │
        ▼
Gate 2: Tech Review — 技术评审（QA + Engineer + PM 三签）
        │
        ▼
QA Case Design（三方：PM + Architect + Engineer 签字）
        │
        ▼
文档 PR（doc-{issue} 分支）— Human Review #1
        │
        ▼
文档 PR 合并 = 设计确认 ✅
        │
        ▼
Gate 3: Implementation — 开发实现（feature-{issue} 分支）
        │
        ▼
Gate 4: QA Validation — 质量验证
        │
        ├── QA 根据 Case Design 开发测试代码（case 代码）
        ├── QA 运行 case 验证 Engineer 完成的代码质量
        ├── 不合格 → 排查问题来源：
        │   • 代码问题 → Engineer 修复 → 重新验证
        │   • Case 问题 → QA 修复 case → 重新验证
        ├── 验证通过 → QA 编写测试报告（QA Report）
        │   • 概述 case 代码逻辑
        │   • 测试情况
        │   • 修复情况（如有）
        │
        ├── PM + Architect + Engineer 三方评审测试报告并签字
        │   • PM: 是否满足 PRD 需求
        │   • Architect: 是否满足 Case Design 和 Tech Spec
        │   • Engineer: 修复情况如何修复的
        │
        ▼ (通过)
代码 PR（feature-{issue} 分支）— Human Review #2
        │
        ▼
代码 PR 合并 = 实现确认 ✅
        │
        ▼
PM 关闭 Issue（通过 github_issue_sync.py 发布关闭请求）
```

#### 流程图 C：需求变更与打回流程

```
任意节点被 Human 打回
        │
        ▼
回滚到被打回节点的编写阶段
        │
        ▼
修改文档（版本+1，如 v1.0 → v1.1）
        │
        ▼
从该节点开始重新走完整流程（正向流程）
        │
        ▼
继续后续节点直到通过

特殊场景：
• PRD 变更 → Tech Spec 需同步变更（版本+1）→ Case Design 需同步变更（版本+1）
• Tech Spec 重大变更 → 回到 Gate 1 重新评审 PRD
• 需求变更 = 回到 PRD 编写流程 → 按顺序重新走完整个流程
```

#### 流程图 B：Agent Team 启动后 Issue 扫描与路由

```
Agent Team 启动
        │
        ▼
扫描所有 open GitHub Issues
        │
        ▼
┌─────────────────────────────────────────────────┐
│  对每个 Issue 根据当前状态继续对应流程节点：      │
│                                                   │
│  • Issue 处于 Gate 1 阶段  →  继续 PRD Review    │
│  • Issue 处于 Gate 2 阶段  →  继续 Tech Review   │
│  • Issue 处于 QA Case Design → 继续 Case 签字   │
│  • Issue 处于文档 PR 阶段  →  继续 HR#1          │
│  • Issue 处于 Implementation → 继续开发         │
│  • Issue 处于 Gate 4 QA   →  继续 QA 验证       │
│  • Issue 处于代码 PR 阶段  →  继续 HR#2          │
│  • Issue 处于 Gate 5 Release → 继续发布         │
│                                                   │
│  Human 随时可能创建新 Issue                       │
│  新 Issue 从"PM 领取"开始，进入流程图 A          │
└─────────────────────────────────────────────────┘
        │
        ▼
Team Lead 协调多角色并行工作
```

**关键区分**：
- 流程图 A 描述单个 Issue 的完整生命周期（从 PM 认领到 PM 关闭）
- 流程图 B 描述 Team 启动后的全局视角：根据所有 Issue 的当前状态分配工作

**注意**：流程图 A 中的"PM 领取 Issue"是每个 Issue 的起点，不代表 Team 每次只处理一个 Issue。Team 可能同时处理多个处于不同阶段的 Issue。

### 3.2 V2.0 核心规则（Critical Rules）

| # | 规则 | 说明 |
|---|------|------|
| 1 | Issue 是强制入口 | 无 Issue 不得进入 PRD |
| 2 | 讨论结果必须 Comment | 每次讨论结束后必须将结论 Comment 到 Issue |
| 3 | 方案不等于有效 | 当 Issue 包含解决方案时，PM 必须回归问题本身讨论 |
| 4 | PRD 不含技术实现 | PRD 只描述问题/产品设计/能力边界/方案，不含技术实现 |
| 5 | PRD Comment 到 Issue | PRD 完成后在 Issue Comment 简要概述并附链接 |
| 6 | **打回 = 回滚 + 重新走流程** | 任意节点被打回 → 回滚到该节点编写阶段 → 修改文档（版本+1）→ 从该节点重新走完整流程 |
| 7 | **文档版本规范** | 每个文档必须包含：版本历史表（v1.0/v1.1）、前序文档追溯链、评审记录表 |
| 8 | **变更级联效应** | PRD 变更 → Tech Spec 版本+1 → Case Design 版本+1；Tech Spec 重大变更 → 回到 Gate 1 重新评审 PRD |
| 9 | **评审过程必须记录** | 每次评审记录：结果（通过/打回）、具体意见、打回原因、修改后版本 |

---

### 3.2.1 文档版本与变更规范

**每个文档（PRD、Tech Spec、Case Design、测试报告）必须包含**：

1. **版本历史表**：
   ```
   | 版本 | 日期 | 变更说明 | 变更人 |
   |------|------|---------|-------|
   | v1.0 | 2026-04-20 | 初始版本 | PM |
   | v1.1 | 2026-04-21 | 需求变更：xxx，评审打回后修改 | PM |
   ```

2. **前序文档追溯链**：
   ```
   PRD v3.1 → Tech Spec v2.3 → Case Design v1.4 → 测试报告 v1.2
   ```
   每个文档必须明确标注依赖的前序文档版本号

3. **评审记录表**：
   ```
   | 评审日期 | 评审节点 | 评审结果 | 评审人 | 意见/打回原因 | 修改后版本 |
   |----------|---------|---------|-------|-------------|-----------|
   | 2026-04-20 | Gate 1 | 打回 | Human | 缺少非目标定义 | v1.1 |
   | 2026-04-21 | Gate 1 | 通过 | PM+Architect+QA | — | v1.1 |
   ```

**变更传播规则**：
- PRD 变更 → Tech Spec 需同步变更（版本+1）→ Case Design 需同步变更（版本+1）
- Tech Spec 重大变更 → 回到 Gate 1 重新评审 PRD
- 任意文档变更 → 对应下游文档版本+1 → 重新评审

**评审过程记录要求**：
- 每次评审必须记录：评审结果（通过/打回）、具体意见、打回原因
- 打回后修改必须记录：修改了什么、为什么这样改
- 重新评审必须记录：相比上次修改了什么、新的评审意见

### 3.3 V2.2 双 PR 机制

**分支命名**：
- 文档 PR：`doc-{issue_number}-{简短描述}`
- 代码 PR：`feature-{issue_number}-{简短描述}`

**文档 PR 创建条件**（必须同时满足）：
```
✅ PRD 文档完成（docs/prd/PRD-XXX_v*.md）
✅ Tech 文档完成（docs/tech/Tech-XXX_v*.md）
✅ Tech Review 两签通过（PM + QA）
✅ QA Case Design 完成（docs/qa/QA-Case-XXX_v*.md）
```

**代码 PR 创建条件**（必须同时满足）：
```
✅ 文档 PR 已合并（设计已确认）
✅ 代码开发完成
✅ QA 测试报告完成（docs/qa_case_report/）
✅ 三方签字验收完成（Architect + Engineer + PM）
```

### 3.4 V2.2.4 Issue Comment 强制要求

| 节点 | 执行者 | Comment 内容 | 示例 |
|------|--------|-------------|------|
| Issue 领取 | PM | 说明领取，开始分析 | "Issue #N 已领取，开始分析需求" |
| PRD 产出 | PM | PRD 链接 + 概述 | "PRD 已完成: [链接] - 概述..." |
| PRD 评审完成 | PM | 评审结论 + 签署人 | "PRD 评审通过 - PM[×] Architect[×] QA[×]" |
| Tech 产出 | Architect | Tech 链接 + 概述 | "Tech 已完成: [链接] - 概述..." |
| Tech Review 完成 | Architect | 评审结论 + 签署人 | "Tech Review 通过 - PM[×] QA[×]" |
| QA Case Design 完成 | QA | Case 链接 + 概述 | "QA Case Design 完成: [链接]" |
| 文档 PR 创建 | PM | PR 链接 + 文档清单 | "文档 PR 已创建: [链接] - PRD/Tech/QA Case" |
| 文档 PR 合并 | PM | 合并确认 | "文档 PR 已合并，设计确认完毕" |
| 开发完成 | Engineer | 开发报告 | "开发完成 - [组件列表] - 待测试" |
| 测试完成 | QA | 测试报告链接 | "测试完成 - [报告链接]" |
| 代码 PR 创建 | Engineer | PR 链接 + 测试报告 | "代码 PR 已创建: [链接] - Fixes #N" |
| 代码 PR 合并 | Engineer | 合并确认 | "代码 PR 已合并 - 等待 Issue 关闭" |
| Issue 关闭 | PM | 关闭原因 | "Issue #N 已关闭 - 完成/重复/无法复现" |

**⚠️ 强制要求**：未 Comment 视为未完成。每个角色完成工作后必须 Comment。

## 七、v1.2 变更详情

### 变更背景

用户指出 Gate 4 QA Validation 描述不够准确，需要明确：
1. QA 根据 Case Design 开发测试代码（case 代码）
2. QA 运行 case 验证 Engineer 完成的代码质量
3. 不合格时排查问题来源（代码问题 → Engineer 修复；Case 问题 → QA 修复）
4. 验证通过后，QA 编写测试报告（QA Report）
5. 测试报告包含：case 代码逻辑概述、测试情况、修复情况
6. PM + Architect + Engineer 三方评审测试报告并签字
7. 签字通过后才能进入 Human Review #2

### 变更内容

流程图 A 中 Gate 4 QA Validation 细化：
```
Gate 4: QA Validation — 质量验证
        │
        ├── QA 根据 Case Design 开发测试代码（case 代码）
        ├── QA 运行 case 验证 Engineer 完成的代码质量
        ├── 不合格 → 排查问题来源：
        │   • 代码问题 → Engineer 修复
        │   • Case 问题 → QA 修复 case
        ├── 验证通过 → QA 编写测试报告（QA Report）
        │   • 概述 case 代码逻辑
        │   • 测试情况
        │   • 修复情况（如有）
        │
        ├── PM + Architect + Engineer 三方评审测试报告并签字
        │   • PM: 是否满足 PRD 需求
        │   • Architect: 是否满足 Case Design 和 Tech Spec
        │   • Engineer: 修复情况如何修复的
        │
        ▼
代码 PR（feature-{issue} 分支）— Human Review #2
```

## 八、v1.3 变更详情

### 变更背景

用户指出：
1. 需要强调 Issue 流程的打回机制，Human review 可能触发打回，如果打回的是 PRD 文档，就回到 Gate 1 之前的 PRD 编写流程
2. 即使没有到 Human Review #1，也可能有临时需求变更，需求变更意味着流程重新回到 PRD 变更，重新评审，按流程重新走
3. Tech Spec、Case Design 同样处理
4. 需要强调文档规范，每次变更都要有版本记录和变更记录
5. Tech Spec、Case Design 等都要明确自己对应的是什么版本的前序文档

### 变更内容

1. **流程图 A 增加打回机制**：
   - 每个 Gate/评审节点增加"如打回"分支
   - 打回后修改对应文档 → 版本+1 → 重新评审

2. **增加打回机制说明**：
   - 任意评审节点可能被 Human 打回
   - PRD 变更 → Tech Spec 需同步变更 → Case Design 需同步变更（级联效应）
   - 需求变更 = 回到 PRD 编写流程 → 按顺序重新评审

3. **新增 3.2.1 文档版本与变更规范**：
   - 版本历史表：每个文档必须包含
   - 前序文档追溯链：PRD → Tech Spec → Case Design → 测试报告
   - 评审记录表：评审日期、结果、评审人、意见/打回原因、修改后版本
   - 变更传播规则
   - 评审过程记录要求
