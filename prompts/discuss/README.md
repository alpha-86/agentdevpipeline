# Discuss 目录

本目录存放带编号的讨论记录、复核结论、差距审计和阶段性改进建议。

## 使用规则

- 文件按严格编号递增，格式：{编号}_{日期}_{标题}.md。例如：003_2026-04-09_agents定义与必读机制对照审计.md
- 一个项目或者plan的子任务文档要复用父项目的编号，不要创建新的编号，采用 {编号}_phase{N}_{日期}_{标题}.md的形式，其中phase编号和标题要严格对应于父项目文档那的phase编号和标题。例如："017_2026-04-10_phase1_1_README与CHANGELOG复核.md"应改为"016_phase_1_1_2026-04-10_README与CHANGELOG复核.md"，因为这是016文档的子phase。
- 如果产生需求变更，需要增加新的编号，并在该文档中明确是哪个文档的需求变更，并明确为新的文档版本，文件名格式 {编号}_{日期}_{标题}_{文档版本}_from_{编号}.md。新的变更后需求，不应该只有变更部分，应该是基于原文档的重新迭代，增加新的版本号，并在文档最开始写清楚变更原因和变更内容。原文档的子phase不再继续，基于新的文档，继续开展子phase，但是不要重复开展任务。
- review/复核/迭代类文档：**不得使用新编号**，必须基于被 review 的文档编号。文件名格式 {被review编号}_{日期}_{被review编号}review_{标题}.md。例如：031 的 codex review 应命名为 `031_2026-04-15_031_codex_review.md`；031 的迭代 review 应命名为 `031_2026-04-15_031迭代review_实现补齐.md`。**新编号仅限全新内容**，不得用于 review 或迭代已有文档。
- 每份讨论记录都应有日期和主题
- 讨论记录不直接替代正式规则文档
- 讨论达成共识后，应继续沉淀到 `prompts/discuss`、`docs/` 或 `skills/shared/`
