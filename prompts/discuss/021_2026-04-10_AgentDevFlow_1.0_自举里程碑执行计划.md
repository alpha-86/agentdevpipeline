在不引入业务语义的前提下，使 AgentDevFlow 在 Claude Code 和 Codex 中能：

**1.0 验收口径：Claude 和 Codex 都能用 AgentDevFlow 完整跑通"Issue → 文档 PR → 代码 PR → 验收"闭环，不需要人工介入规则层面。**

## Phase 1：自动化层基础建设（最优先）

### 1.1 GitHub Issue 自动化层

**迁移 `scripts/github_issue_sync.py` 及相关 workflow**

**重要：AgentDevFlow 自身必须有 `.github/workflows/` 目录才能自举，使用者项目也需要复制这些文件。**

目标文件：
```
AgentDevFlow 自身目录：
.github/workflows/
  post-comment.yml       # 从 hedge-ai 迁移
  issue-status-sync.yml  # 从 hedge-ai 迁移
  doc-pr-checks.yml      # 从 hedge-ai 迁移
  code-pr-checks.yml     # 从 hedge-ai 迁移

scripts/
  github_issue_sync.py    # 从 hedge-ai 迁移，需剥离业务语义

使用者项目复制：
.github/workflows/        # 从 AgentDevFlow 模板复制
```

关键改造点：
- 移除所有业务相关逻辑（验证、指标、方案等）
- Token 加载逻辑保留（从 `~/.env.code` 读取）
- TaskQueue 格式保留，但任务路由改为 AgentDevFlow 的角色体系
- Comment 格式改为 AgentDevFlow 的 PRD/Tech/QA/验收标准

交付物：`scripts/github_issue_sync.py` 及 4 个 GitHub Actions workflow

### 1.2 Telegram 通信 channel

**迁移 `hedge_ai/telegram/` 和 `scripts/util/send_telegram.py`**

目标文件：
```
scripts/
  send_telegram.py       # 迁移（通用消息推送）
  telegram_bot.service    # 迁移（systemd service）

hedge_ai/
  telegram/
    bot_server.py        # 迁移（需剥离业务语义）
  core/
    telegram_monitor.py   # 迁移（通用通知监控）

.github/workflows/
  telegram-alerts.yml     # 新增（如果需要定时通知）
```

关键改造点：
- 移除 `bot_server.py` 中的业务信号推送、方案推送等业务逻辑
- 保留通用的消息收发、健康检查、重试机制
- `send_telegram.py` 保留为通用 CLI 工具
- Telegram channel 的使用场景是"Agent 向用户推送状态通知"，不是业务告警

交付物：Telegram Bot 服务器框架 + 消息发送脚本

### 1.3 TaskQueue 运行时机制

**设计并实现 AgentDevFlow 的任务分配运行时**

**重要：必须用文件方式（`.claude/task_queue/*.task`），Claude Code Agent Team 可识别，不重复造轮子。**

目标文件：
```
.claude/
  task_queue/             # 任务队列目录（新建）
  results/                # 结果目录（新建）

scripts/
  task_router.py         # 新增（Issue → Task 路由）
  task_update.py         # 新增（Task 状态更新）
```

关键设计点：
- Task 文件格式：`source=agentdevflow`，按 AgentDevFlow 角色路由
- 路由规则：`type/prd` → PM，`type/tech` → 架构师，`type/bug` → Engineer，等
- AI 驱动 AI：Agent 启动时自动检查 TaskQueue，不等 Team Lead 触发
- Team Lead 只分配（TaskUpdate 设 owner），不 trigger 执行

交付物：TaskQueue 运行时脚本

## 与 hedge-ai 专家对齐结论

通过与 hedge-ai V3.0 prompts 专家和 .claude/skills 专家的讨论，确认以下关键发现：

### 1. 核心缺失：三层保障体系的执行层不存在

hedge-ai 的流程强制执行靠**三层保障体系**：
1. **GitHub 原生能力**：Branch Protection + PR Template + Merge Gate
2. **CI/CD 自动检查**：4 个 GitHub Actions workflow
3. **Agent Critical Rules**：规则写在 agent 定义里

AgentDevFlow **只有第三层且只是 Markdown**，没有 CI enforcement。

### 2. AI 驱动 AI：事件总线 vs TaskQueue

- **事件总线**：hedge-ai 的 AI 驱动 AI 主要靠事件总线（策略事件触发→评价网络→决策）
- **TaskQueue**：主要用于 GitHub Issue 驱动工作流，Agent 领取自己类型的 Task
- **AgentDevFlow 缺失**：没有事件总线，没有 TaskQueue 运行时

### 3. 最小必须迁移的可执行组件（两个专家共识）

| 优先级 | 组件 | 说明 |
|---|---|---|
| P0 | `.github/workflows/doc-pr-checks.yml` | 文档 PR 完整性检查 |
| P0 | `.github/workflows/code-pr-checks.yml` | 代码 PR 前置条件检查 |
| P0 | `.github/workflows/issue-status-sync.yml` | Issue 状态同步 |
| P0 | `scripts/github_issue_sync.py` | 统一 Comment 发布入口 |
| P1 | Team Lead 任务分配检查清单自动化 | CI 检查文档 PR 已合并 |
| P1 | 交付物合规检查 CI | PRD/Tech/QA Case 目录和命名规范 |
| P2 | Telegram Bot 框架（通用 channel） | 通知用户 Agent 状态 |

### 4. Issue Comment 强制要求（V2.2.4）

hedge-ai 要求每个关键节点必须 Comment，未 Comment 视为未完成。必须通过 `scripts/github_issue_sync.py` 发布，禁止直接用 `gh issue comment`。
