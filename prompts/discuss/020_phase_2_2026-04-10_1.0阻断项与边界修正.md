# 020 Phase 2 - 1.0 阻断项与边界修正

## 目的

本文件用于在进入平台对等方案设计之前，先锁定会改变 1.0 边界的阻断项，避免后续 Claude / Codex 方案基于错误前提定稿。

## 结论摘要

本阶段确认以下 4 个边界决策，后续所有平台方案和实现都必须服从这些决策：

1. **对外独立性必须落地**
   - 面向使用者的入口文档，不再把 `hedge-ai` 作为理解前提。
2. **角色体系必须调整**
   - `Tech Lead` 改为 `架构师`
   - `Researcher` 改为可选支持角色，而不是默认必选角色
3. **复用原则提升为正式原则**
   - 优先复用 Claude / Codex / GitHub / `gh` / CI / 现成 skills 与 plugins 的已有能力，避免重复造轮子。
4. **004 的 1.0 边界需要收紧**
   - GitHub Issue / PR / Comment 主线是 1.0 必需项
   - “评论真实送达失败即阻断”是 1.0 必需项
   - Telegram 自动通知、独立消息桥、cron 自动投递不是 1.0 必需项，可作为后续增强

## 一、独立性边界修正

### 当前问题

根 `README.md`、`plugins/agentdevflow/README.md`、部分平台接入文档和角色说明，仍然把：

- “从 hedge-ai 拆出来”
- “对照 hedge-ai”
- “必须回溯到 hedge-ai 源文件”

当作面向使用者的主解释逻辑。

这与 `prompts/by-me/003_本项目独立性的说明.md` 冲突。

### 决策

区分两类文档：

1. **使用者入口文档**
   - `README.md`
   - `plugins/agentdevflow/README.md`
   - `docs/platforms/*.md`
   - `adapters/*/README.md`

这些文档必须独立自洽，解释：

- AgentDevFlow 解决什么问题
- 为什么这么组织
- 如何接入和运行

但不再要求读者先理解 `hedge-ai`。

2. **内部审计与迁移文档**
   - `docs/migration/*`
   - `docs/governance/migration-boundary-from-hedge-ai.md`
   - `prompts/discuss/*`

这些文档可以继续保留 `hedge-ai` 引用，因为它们服务的是内部追溯和边界审计。

### 对后续的影响

- `README.md` 与插件入口必须重写对外叙述主线
- 平台接入文档必须以“如何启动 AgentDevFlow”来写，而不是“它来自哪里”

## 二、角色体系修正

### 当前问题

当前仓库仍然大量保留：

- `Tech Lead`
- `tech-lead.md`
- `Researcher`
- `researcher.md`
- 平台文档把 `Researcher` 写成默认建议角色
- 团队启动顺序把研究角色放进默认装载列表

### 决策

#### 1. `Tech Lead` 改为 `架构师`

后续统一规则：

- 中文正式展示名统一为：`架构师`
- 若需要兼容过渡，可在内部迁移说明中短暂保留“原 `Tech Lead`”提示
- 用户入口文档、平台文档、团队拓扑、角色规范，均不得继续把 `Tech Lead` 作为主显示名

#### 2. `Researcher` 改为可选角色

后续统一规则：

- 研究角色保留，但定位为“按需启用的支持角色”
- 默认最小团队不再把研究角色列为必选
- 只有在需求存在高不确定性、需要额外调研时，才显式启用

### 对后续的影响

- `001_team_topology.md`
- `002_product_engineering_roles.md`

- `skills/shared/create-agent.md`

- `README.md`
- `plugins/agentdevflow/README.md`

这些文件都需要在 Phase 3 中做统一改名与默认角色修正。

## 三、复用原则正式化

### 当前问题

仓库里已经多处提到：

- 优先复用共享入口
- 外部依赖优先引用和适配
- 避免重复造轮子

但它还没有成为 1.0 的正式决策主线，导致 `004` 等需求容易被理解成“要把所有脚本和消息机制都自己重做一遍”。

### 决策

新增正式原则：

> AgentDevFlow 在 1.0 阶段必须优先复用现成能力，避免重复造轮子。  
> 只要 Claude、Codex、GitHub、`gh`、CI、已有 skills / plugins / agents 包已经能承载目标流程，AgentDevFlow 就只补规则约束、入口组织和缺失的最小资产，不重复实现宿主能力。

### 复用优先级

1. 宿主平台原生能力
   - Claude Agent Team
   - Codex 多 agent
   - GitHub Issue / PR
2. 通用协作工具
   - `git`
   - `gh`
   - CI
3. 可选增强宿主
   - `gstack`
   - `superpowers`
4. 只有以上都无法满足时，才新增 AgentDevFlow 自身承载

### 对后续的影响

- `docs/governance/core-principles.md` 需要新增该原则
- `docs/reference/dependencies.md` 需要把“必需 / 推荐 / 可选”与复用策略统一起来
- Claude / Codex 方案不得再重复造 Team、消息、PR、Issue 基础能力

## 四、004 的 1.0 边界判定

### 当前问题

`004` 原始要求写得很大，覆盖：

- GitHub Issue / PR
- 消息通信机制
- Telegram
- prompt / workflow / python / cron 脚本

如果不先压缩到 1.0 自举所需最小闭环，后续实现会失控。

### 决策：1.0 必需项

以下能力属于 **1.0 必需项**：

1. GitHub Issue 作为主流程索引
2. 双阶段 PR 与 Issue 回链规则
3. Issue Comment Gate
4. 评论真实送达失败必须阻断，而不能伪装成功
5. 有一条可执行的评论回写路径
   - 可以是宿主平台能力
   - 可以是 `gh`
   - 可以是人工执行但必须有明确步骤和失败处理
6. 人审、留痕、状态同步都能通过 Issue / PR / 文档被反查

### 决策：1.0 可后置项

以下能力 **不作为 1.0 达成的前置条件**：

1. Telegram 自动通知
2. 独立的消息桥服务
3. cron 自动补发或巡检脚本
4. 专门为某个平台单独重写一整套通信层

### 理由

- 这些能力能增强效率，但不是 1.0 自举闭环的最小必要条件
- 若宿主平台已有能力可承载，就不应为了“迁移完整度”重复造轮子
- 1.0 的目标是“真实跑通 AgentDevFlow 自举”，不是“复刻 hedge-ai 的所有基础设施”

### 对后续的影响

- Phase 3 需要把“支持的评论回写路径”写清楚
- Phase 4/5/6 的演练中必须实际证明评论回写与失败阻断可执行
- Telegram 与独立消息层进入 1.0 之后的 backlog

## 六、Phase 2 最终结论

`Phase 2` 到此完成，后续进入 `Phase 3` 时必须基于以下前提：

1. 平台方案只面向“独立可理解的 AgentDevFlow”，不再以 `hedge-ai` 作为用户理解前提
2. 角色体系以 `架构师` + “研究角色可选” 为准
3. 复用现成能力是正式原则，不再默认自建底层能力
4. `004` 中真正阻断 1.0 的，是 Issue / PR / Comment / 失败阻断闭环，不是 Telegram 自动化
