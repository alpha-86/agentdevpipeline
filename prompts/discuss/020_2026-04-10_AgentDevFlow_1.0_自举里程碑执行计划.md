# 020 AgentDevFlow 1.0 自举里程碑执行计划

## 文档定位

本文件用于承接 `prompts/by-me/005_AgentDevFlow_1.0_自举_里程碑目标.md`，作为 AgentDevFlow 1.0 的主计划文件。

本计划不沿用“001、002、016 已经完成即可视为 1.0 基本达成”的前提，而是重新以“真实可自举”为唯一验收口径做整体验真、补齐、演练和收口。

## 目标

在不突破 `hedge-ai` 可迁移研发流程边界的前提下，使 AgentDevFlow 在 `Claude Code` 与 `Codex` 中都能真实完成一轮“使用 AgentDevFlow 迭代 AgentDevFlow 自身”的闭环，从而达成 1.0 的关键里程碑：`自举`。

## 1.0 严格验收口径

1. 不以文档“已写完”作为完成标准，必须以真实执行证据作为完成标准。
2. `Claude Code` 侧必须基于现有 Agent Team 能力完成一轮自举演练，不重复建设平台已经具备的 Team 机制。
3. `Codex` 侧必须基于现有多 agent 能力完成一轮对等自举演练，允许复用平台能力，例如 `spawn_agent / send_input / wait_agent` 一类机制。
4. 两个平台都必须形成明确的：
   - 启动入口
   - 角色装载规则
   - 工作流执行顺序
   - 交付物落点
   - 人机协同 Gate
   - 演练记录与结论
5. 自举演练必须针对 `AgentDevFlow` 仓库自身，而不是虚构 demo 项目。
6. 1.0 能力边界必须等同于 `hedge-ai` 中可迁移的研发全流程机制边界，不能额外创造 `hedge-ai` 没有的新核心能力。

## 核心原则

### 1. 真实验真优先

- 不接受“应该能跑”“理论可用”“文档已覆盖”作为 1.0 达成依据。
- 任何主张都必须能落到实际平台入口、执行过程和交付证据。

### 2. 对等复用优先，避免重复造轮子

- AgentDevFlow 必须优先复用现成能力，避免重复实现平台已经提供或社区已有成熟承载的能力。
- 可复用对象包括但不限于：
  - Claude / Codex 已有能力
  - 已有 agent team / multi-agent 编排能力
  - 已有 skills 包、plugin、agents 包
  - 已有开源流程资产与工具脚本
- 只有当现成能力无法满足 `hedge-ai` 可迁移机制要求时，才允许新增 AgentDevFlow 自身承载。
- 所有复用决策都必须记录：
  - 复用了什么
  - 为什么可复用
  - 哪些部分仍需 AgentDevFlow 自己补

### 3. 严守迁移边界

- 1.0 只迁移 `hedge-ai` 中可泛化到通用软件研发的机制。
- 禁止引入量化交易业务语义，也禁止借 1.0 自举目标扩展出源项目没有的新主流程。

### 4. 人机协同不可降级

- 双阶段 PR、Issue 驱动、Review Gate、交付留痕、异常升级、恢复机制，仍然是 1.0 主干，不因平台差异而省略。

## 当前判断

### 已知事实

- `003`、`004` 对应的能力，用户已明确说明“还没实现”，不能跳过。
- 仓库已有较多迁移和审计文档，但这些文档本身不足以证明 1.0 自举已经达成。
- 当前 `docs/platforms/claude-code.md` 与 `docs/platforms/codex.md` 仍偏“接入说明”，还不是“可验收的自举运行方案”。
- 当前 `docs/migration/completion-audit.md` 与 `docs/migration/review-vs-hedge-ai.md` 的完成度表述，对 1.0 自举目标来说仍偏乐观。

### 当前主风险

1. 把“文档迁移完成”误当成“平台可自举”。
2. Claude / Codex 的对等机制没有真正拆到可执行层。
3. `001`、`002` 中已有结论可能存在漏项或错误抽象。
4. `003`、`004` 涉及的独立性、Issue/消息能力若未补齐，会直接阻断 1.0 自举。

### 可并行部分

- `Phase 1` 中：
  - `001~005` 需求验真
  - 当前仓库入口文档验真
  - `hedge-ai` 源机制抽取
  这三块可以并行
- `Phase 2` 中：
  - 独立性口径修正
  - 复用策略与依赖说明
  - Issue / 消息机制 1.0 边界澄清
  可以分工并行
- `Phase 3` 中：
  - Claude 对等方案
  - Codex 对等方案
  可以并行，但最后必须回到统一主干对齐
- `Phase 5` 与 `Phase 6` 不能完全并行开始
  因为 Phase 4 统一验收方案先要锁定
  但进入执行后，Claude / Codex 的具体验证与修补可以交叉推进

## subagent 拆分原则

后续进入执行阶段时，建议最少拆成以下几个工作流：

1. `审计组`
   - 负责 `001~005`、`016/019`、平台文档、hedge-ai 源的差距核对
2. `治理与边界组`
   - 负责独立性口径、复用原则、Issue/消息边界和完成度口径修正
3. `Claude 适配组`
   - 负责 Claude Team 入口、角色必读、运行链路与演练
4. `Codex 适配组`
   - 负责 Codex 多 agent 对等编排、插件入口与演练
5. `治理与文档组`
   - 负责 README、平台文档、依赖说明和验收文档

要求：

- 每个 subagent 必须有清晰写入范围，避免互相覆盖。
- 审计结果与阻断项修正先于平台方案定稿。
- 任何实现都必须回挂到 1.0 验收口径，不做与目标无关的“顺手优化”。

## 本计划与已有 016/019 的关系

- `016` 解决的是“从 hedge-ai 拆出独立插件资产并完成中文主线收口”。
- `020` 解决的是“在严格 1.0 口径下，证明 AgentDevFlow 已具备真实自举能力”。
- 因此 `020` 不是 `016` 的简单续 phase，而是新的里程碑主计划。
- `016/019` 中已有结论可以复用，但必须接受 `020` 的重新验真，不得直接继承为完成事实。

## 当前讨论结论

1. 1.0 验收口径采用最严格标准：Claude 与 Codex 都必须真实跑通一轮自举。
2. Codex 允许依赖平台已有多 agent 能力实现对等编排。
3. AgentDevFlow 必须新增并固化一条重要原则：`优先复用现成能力，避免重复造轮子`。
4. Phase 顺序调整为：先验真，再修正阻断项与边界，再锁平台对等方案，再演练、再验收。
5. 当前优先路线采用“验证驱动型”，先验真、再补齐、再演练、再验收。
