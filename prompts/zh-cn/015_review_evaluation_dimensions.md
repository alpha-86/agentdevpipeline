# 评审评估维度

## 目标

把评审从“主观感觉”收敛为统一的检查维度，适用于 PRD、Tech、QA、Release 与流程合规审计。

## 四个核心维度

### 1. 完整性

- 输入是否完整
- 必填字段是否齐全
- 下游执行所需信息是否具备

### 2. 一致性

- PRD / Tech / QA / Release 之间是否一致
- 文档状态、Issue 状态、Todo 状态是否一致
- 评论结论与正式文档是否一致

### 3. 逻辑性

- 目标、方案、验收之间是否逻辑闭环
- 风险与结论是否匹配
- 变更级别与重审动作是否匹配

### 4. 证据充分性

- 是否有可验证链接或记录
- 结论是否有测试、评审、发布等证据支撑
- 是否只是口头说明而无正式留痕

## 使用规则

- 任一维度明显不满足时，不得给出 `Approved`。
- `Conditional` 必须明确缺的是哪一维、由谁补。
- `Rejected` 必须指出最关键的失败维度。

## 典型失败模式

- 文档看起来完整，但与 Issue 状态不一致
- 测试报告存在，但无法证明覆盖关键验收项
- 结论合理，但没有证据链接
- 方案逻辑自洽，但与上游需求目标不一致
