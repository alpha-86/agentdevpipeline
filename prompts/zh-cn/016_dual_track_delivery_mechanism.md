# 双轨交付机制

## 目标

迁移 hedge-ai 中“双 PR 思想”的通用部分，形成适用于本项目的“双轨交付机制”，把“流程确认”和“实现交付”分离，降低流程被代码推进裹挟的风险。

## 双轨定义

### 轨道 A：流程与文档轨

- PRD
- Tech Spec
- Change Record
- Review Memo
- QA 结论
- Release Record

### 轨道 B：实现与交付轨

- 代码实现
- 自动化测试
- 配置变更
- 部署与发布执行

## 核心原则

- 轨道 B 不得在轨道 A 缺关键批准时强行推进。
- 轨道 A 的变更若达到 Major / Breaking，轨道 B 必须等待重审。
- 轨道 A 与轨道 B 都必须回链同一个主 Issue。

## 适用场景

- 需求和实现并行推进但需要防止范围漂移
- 代码已准备好，但流程证据还未完成
- 需要把“实现完成”与“可发布”明确区分

## 执行规则

1. 轨道 A 先建立主文档与 Gate 结论。
2. 轨道 B 在已有批准范围内推进实现。
3. 若轨道 A 发生 Major / Breaking 变化，轨道 B 暂停并回看影响。
4. 合并或发布前，必须再次校验两条轨道状态一致。

## 禁止行为

- 用“代码已经写完”为理由绕过文档 Gate
- 文档已变更但实现轨道不回看
- 只在代码轨道留痕，不在流程轨道留痕
