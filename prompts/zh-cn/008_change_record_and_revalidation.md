# 变更记录与重审机制

## 目标

保证任何中后期变更都能被正确分级、记录、重审，防止“文档与实现漂移”。

## 变更分级

### Minor

- 不改变需求边界和核心行为
- 仅影响表述、非关键实现细节、低风险修正

处理要求：

- 更新相关文档
- 在 Review Record 追加记录
- 不强制回到上游 Gate

### Major

- 影响关键流程、接口、数据结构或测试路径
- 可能改变验收方式或风险画像

处理要求：

- 必须更新 PRD/Tech/QA 至一致状态
- 必须触发对应 Gate 重审
- Issue 中记录“变更原因 + 影响分析 + 重审结论”

### Breaking

- 改变需求目标、业务边界或上线策略
- 造成跨角色职责、排期、发布计划显著变化

处理要求：

- 回退至受影响最上游 Gate 重新评审
- Team Lead 发起升级评审
- 未完成重审前禁止继续下游阶段

## Change Record 最小字段

- 变更 ID
- 变更级别（Minor/Major/Breaking）
- 提出人
- 变更原因
- 影响范围（PRD/Tech/QA/Release/Todo）
- 需要重审的 Gate
- 决策人
- 决策日期

## 一致性校验

每次变更后必须校验：

1. PRD 与 Tech 是否仍一一映射。
2. QA Case 是否覆盖变更后的关键路径。
3. Todo 是否新增或更新了重审动作。
4. Release 说明是否同步风险变化。

任一失败，则变更不得标记完成。

## 违规处理

- 发现“先改实现后补文档”：标记流程违规并回到相应 Gate。
- 发现“只改文档不改测试”：阻断进入 Release Gate。
- 发现“跳过变更分级”：按 `Major` 默认处理并补齐记录。
