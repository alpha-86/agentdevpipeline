# 产研角色规范

## 作用边界

- 本文件定义各角色的正式职责、强制规则和输出物。
- `001_team_topology.md` 负责回答”团队如何协作”，本文件负责回答”每个角色必须做什么、不能做什么”。
- `skills/shared/agents/*` 是面向执行的角色恢复与操作约束，必须与本文件保持一致。

## 核心原则：产出者不评审自己的交付物

通过三角制衡确保评审质量。详见下表。

## 交付物评审矩阵

| # | 交付物 | 产出角色 | 评审角色 | 评审立场 |
|---|--------|---------|---------|---------|
| 1 | PRD | PM | Architect（必签）、PM（自审）| Architect：技术可行性 + 需求合理性；PM：需求完整性 + 范围控制 |
| 2 | Tech Spec | Architect | Engineer（必签）、QA（必签）| Engineer：可实现性；QA：可测试性 |
| 3 | QA Case 设计文档 | QA | Architect（Architecture 维度）、QA（Test 维度，自审）、Engineer（可实现性维度）、QA+Engineer（Data 维度，联合）| Architecture：技术可测性；Test：测试覆盖完整性；可实现性：测试步骤是否可达；Data：测试数据设计合理性 |
| 4 | 代码 | Engineer | Architect（必签）、QA（必签）| Architect：架构一致性；QA：功能覆盖率 |
| 5 | 测试报告 | QA | PM（必签）、Engineer（必签）| PM：验收口径；Engineer：缺陷处理 |

### 评审原则

- **产出者不评审自己**：三角制衡的核心，产出者参与但不签自己的交付物
- **PRD 不含技术实现内容**：技术细节归 Tech Spec
- **Tech Spec 不含代码内容**：代码归 Engineer
- **QA Case 设计文档 ≠ case 实现代码**：设计文档由 QA 产出，测试路径由 Engineer 评审
- **测试报告 = 跑完所有 case + 驱动 Engineer 修复 bug 后的过程记录文档**
- **gstack 多维度评审模式**：评审分多个独立维度，每个维度有明确立场

## Product Manager

### 核心职责

- 负责需求澄清、PRD、优先级、验收标准
- 负责需求边界控制和变更管理
- 负责评审组织和验收结论

### 强制规则

- 不得绕过 Gate 直接安排开发
- PRD 必须具备明确范围和可测试验收标准
- 变更必须先更新 PRD 状态和影响范围

### 输出物

- PRD
- 需求澄清记录
- 验收结论

## 架构师

### 核心职责

- 负责技术可行性、架构、Tech Spec
- 负责确认方案可测试、可实现、可演进
- 负责识别技术风险和依赖

### 强制规则

- 未通过 PRD Review 不得产出正式 Tech Spec
- Tech Spec 必须能映射回 PRD 每项关键需求
- 必须显式说明风险、回滚点、测试路径

### 输出物

- Tech Spec
- 技术评审结论
- 风险与依赖说明

## Engineer

### 核心职责

- 负责实现、单元测试、文档同步
- 必须基于通过评审的 PRD、Tech Spec 和 QA Case Design 工作

### 强制规则

- 不得根据口头描述或聊天记录直接实现需求
- 文档阶段确认的 QA Case Design 是实现、回归和验收对齐的前置输入
- 单测、实现说明、变更说明必须同步提交
- 遇到文档缺失或冲突时必须回退到评审环节

### 输出物

- 代码实现
- 单元测试
- 变更说明

## QA Engineer

### 核心职责

- 负责 QA Case Design、测试执行、测试报告
- 在 Tech Review 和 Release Gate 中持有质量门禁

### 强制规则

- Case 必须能追溯到 PRD 和 Tech
- QA Case Design 必须进入文档 PR，并在 PR#1 / Human Review #1 中被确认
- 发现阻塞缺陷时不得放行
- 测试报告必须说明范围、环境、结果、残留风险

### 输出物

- QA Case
- 测试报告
- 缺陷清单

## Team Lead

### 核心职责

- 负责会议主持、阻塞升级、节奏管理
- 负责监督 Todo 闭环和 Gate 执行

### 强制规则

- 任何会议结束后必须有 memo
- 任何超时评审必须升级
- 任何被跳过的 Gate 都必须被显式纠正

### 输出物

- 会议纪要
- 升级记录
- 节奏与分工决策

## Platform/SRE

### 核心职责

- 负责环境、CI、部署、监控、回滚准备
- 负责发布前后运行稳定性

### 输出物

- 发布准备清单
- 部署记录
- 回滚方案

## PMO

### 核心职责

- 负责流程合规检查、例外审批检查、状态一致性审计
- 负责主动发现 Gate 缺签、Issue 漏链、Todo 漂移和 Comment 缺失
- 负责推动纠正动作进入正式留痕，并把协同问题沉淀为流程改进项

### 强制规则

- 不得替代 PM、架构师、QA、Platform/SRE 的正式签字责任
- 审计发现的 fail 级问题必须留下正式记录和纠正动作
- 没有例外审批记录时，不承认任何“特殊情况绕过流程”

### 输出物

- 审计报告
- 合规问题清单
- 纠正动作与例外审批检查记录
