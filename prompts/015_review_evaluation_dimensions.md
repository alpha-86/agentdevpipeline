# 评审评估维度

## 目标

把评审从“主观感觉”收敛为统一的检查维度，适用于 PRD、Tech、QA、Release 与流程合规审计。

## 四个核心维度

### 1. 完整性

- 输入是否完整
- 必填字段是否齐全
- 下游执行所需信息是否具备

### 2. 一致性

- PRD / Tech / QA / Release 之间是否一致
- 文档状态、Issue 状态、Todo 状态是否一致
- 评论结论与正式文档是否一致

### 3. 逻辑性

- 目标、方案、验收之间是否逻辑闭环
- 风险与结论是否匹配
- 变更级别与重审动作是否匹配

### 4. 证据充分性

- 是否有可验证链接或记录
- 结论是否有测试、评审、发布等证据支撑
- 是否只是口头说明而无正式留痕

## 推荐检查器类型

### 1. 完整性检查器

- 检查输入是否齐全
- 检查必填字段是否缺失
- 检查下游执行所需信息是否完整

### 2. 一致性检查器

- 检查 PRD / Tech / QA / Release 是否互相一致
- 检查 Issue / Gate / Todo / Comment 状态是否一致
- 检查 PR、文档、评论和产物关联是否一致

### 3. 逻辑性检查器

- 检查问题、目标、方案、验收是否闭环
- 检查风险、结论、回退条件是否匹配
- 检查变更级别与重审动作是否匹配

### 4. 证据充分性检查器

- 检查证据链接是否真实存在
- 检查测试、评审、发布证据是否支持当前结论
- 检查是否仍停留在聊天或口头结论

## 检查结论规则

- 四个维度任一项明显失败时，不得给出 `Approved`。
- 若仅存在可补齐项，可给出 `Conditional`，但必须写明：
  - 缺失维度
  - 修复 owner
  - 重试条件
- 若存在阻断性失败，必须给出 `Rejected` 或 `fail`，并阻断下游。

## 最小输出格式

- 检查对象
- 检查器类型
- 结论
- 失败维度
- 关键证据链接
- 修复动作
- 是否允许进入下游阶段

## 使用规则

- 任一维度明显不满足时，不得给出 `Approved`。
- `Conditional` 必须明确缺的是哪一维、由谁补。
- `Rejected` 必须指出最关键的失败维度。

## 典型失败模式

- 文档看起来完整，但与 Issue 状态不一致
- 测试报告存在，但无法证明覆盖关键验收项
- 结论合理，但没有证据链接
- 方案逻辑自洽，但与上游需求目标不一致
- 产物和评论都存在，但无法反查到同一个主 Issue
