# AgentDevPipeline 核心原则

## 目的

本文件定义 AgentDevPipeline 的顶层方法论。后续所有流程演进、角色设计、模板补充、平台适配，都必须与这些原则保持一致。

## 原则 1：Issue First

- Issue 不是附属记录，而是整个交付流程的主索引。
- 一个需求、一个问题、一次流程异常，都必须有可追溯的 Issue 主线。
- PRD、Tech、QA、Release、Todo、Memo、Change Record、Review Comment 都必须回链到 Issue。

这条原则解决的是：

- 多项目并行时如何区分主线
- 一个项目中多个问题如何独立追踪
- Agent 切换、会话中断后如何恢复上下文

## 原则 2：Human In The Loop

- AgentDevPipeline 不只是自动化编排系统，还必须是人机协同系统。
- 关键交付物不能只由 Agent 内部流转，必须在关键 Gate 引入 Human Review 或 Human Confirm。
- Human Review 的对象、时机、形式、通过条件，都必须显式定义，而不是默认口头确认。

最关键的 Human Review 节点通常包括：

- 文档设计确认
- 代码变更确认
- 发布放行确认
- 重大变更与例外审批确认

## 原则 3：设计先于实现

- 没有被确认的设计，不得进入正式开发。
- 需求文档、技术方案、测试设计必须先形成正式交付物，再进入实现阶段。
- “先写代码后补文档”只会制造漂移，不属于允许路径。

## 原则 4：双阶段交付

- 文档确认和代码交付是两个不同阶段，不能混在一个含糊流程里。
- 至少要能够区分：
  - 设计何时被确认
  - 实现何时被确认
- 对于高风险项目，应该采用文档阶段与代码阶段分离的人机协同机制。

## 原则 5：正式留痕优先于聊天推进

- 聊天可以帮助澄清，但不能代替正式交付物。
- 所有关键结论都必须进入文档、Issue 评论、Memo、Todo、Change Record 或 Release Record。
- 没有正式留痕的结论，默认不具备流程效力。

## 原则 6：Gate 不可绕过

- 每个阶段都必须有清晰的输入、输出、签字人、退回条件。
- 任一 Gate 未通过，不得直接进入下游阶段。
- 如果上游发生 Major / Breaking 变更，下游必须回退重审。

## 原则 7：恢复优先于继续执行

- 一旦会话清空、角色切换、执行中断，必须先恢复上下文，再继续动作。
- 恢复时必须先确认当前 Issue、当前 Gate、最新决策、未完成 Todo、已知风险。

## 原则 8：合规要可审计

- 流程规则不能只写在文档里，还要能被检查、被发现、被纠正。
- Team Lead 或治理角色必须能够发现：
  - Gate 被跳过
  - 文档状态漂移
  - Todo 无 owner / due / evidence
  - 异常未升级

## 原则 9：平台适配不能改写核心语义

- Claude、Codex、OpenCode 只是接入入口。
- 平台层可以包装，不可以篡改共享角色、流程、模板的含义。
- 任何平台特性都必须服从共享流程，而不是反过来重写流程。

## 原则 10：坚持通用研发语义

- 本项目只沉淀通用产研流程机制。
- 不引入量化交易组织角色、交易时段、交易日报、回测因子等业务语义。
- 从 hedge-ai 迁移时，只抽方法，不带业务域内容。

## 使用方式

- 新增流程机制前，先检查是否违反上述原则。
- 若需要引入例外，必须形成单独的例外审批记录。
- 若原则之间出现冲突，以 `Issue First`、`Human In The Loop`、`Gate 不可绕过` 优先。
